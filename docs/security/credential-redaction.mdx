<What>

## What is Credential Redaction?

Credential redaction automatically strips known API key and secret patterns from text before it enters the conversation context or is returned to callers. This prevents accidental credential leakage when tool results or error messages contain sensitive values — for example, when a shell command echoes an environment variable containing an API key.

The sanitizer recognizes 14 credential pattern families by their well-known prefixes:

| Pattern | Label |
|---------|-------|
| `ghp_`, `gho_`, `ghs_`, `ghu_`, `github_pat_` | `github-pat`, `github-oauth`, `github-app`, `github-user` |
| `sk-` (20+ chars), `sk-ant-` | `api-key`, `anthropic-key` |
| `AKIA` (16+ chars) | `aws-key` |
| `sk_live_`, `sk_test_`, `pk_live_`, `pk_test_` | `stripe-key` |
| `xoxb-`, `xoxp-`, `xoxs-` | `slack-token` |
| `Bot ` (50+ chars after) | `discord-token` |
| `npm_` (32+ chars) | `npm-token` |
| `SG.` (40+ chars) | `sendgrid-key` |
| `Bearer ` (10+ chars) | `bearer-token` |
| `-----BEGIN ... PRIVATE KEY-----` | `private-key` |

Detected credentials are replaced with `[REDACTED:<label>]`. The original text is never logged.

</What>

<How>

## API

```rust
use genesis_core::sanitize::{sanitize_credentials, contains_credentials};

// Fast check (no allocation)
if contains_credentials(&tool_output) {
    let safe = sanitize_credentials(&tool_output);
    // use `safe` instead of `tool_output`
}

// Or sanitize unconditionally
let safe = sanitize_credentials(&text);
```

## Examples

```rust
// GitHub PAT
sanitize_credentials("token: ghp_ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmn")
// → "token: [REDACTED:github-pat]"

// OpenAI API key
sanitize_credentials("key=sk-proj-1234567890abcdefghijklmno")
// → "key=[REDACTED:api-key]"

// AWS access key
sanitize_credentials("aws_access_key_id = AKIAIOSFODNN7EXAMPLE")
// → "aws_access_key_id = [REDACTED:aws-key]"

// Bearer token in Authorization header
sanitize_credentials("Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9....")
// → "Authorization: [REDACTED:bearer-token]"

// PEM private key
sanitize_credentials("-----BEGIN RSA PRIVATE KEY-----\nMIIE...\n-----END RSA PRIVATE KEY-----\n")
// → "[REDACTED:private-key]"

// Multiple credentials in one string — all redacted
sanitize_credentials("ghp_abc...xyz and sk-proj-123...xyz")
// → "[REDACTED:github-pat] and [REDACTED:api-key]"

// Short prefix without enough suffix — NOT redacted (avoids false positives)
sanitize_credentials("sk-short")
// → "sk-short"
```

## Minimum Suffix Length

Each pattern requires a minimum number of alphanumeric characters after the prefix before it is considered a real credential. This prevents false positives on abbreviated identifiers:

- `sk-` requires 20+ chars after the prefix
- `ghp_` requires 36+ chars
- `AKIA` requires 16+ chars

</How>

<Why>

## Design Decisions

**Prefix + minimum length matching.** Most secrets have well-known, vendor-specific prefixes. Matching on prefix and requiring a minimum suffix length strikes a good balance between recall (catching real secrets) and precision (avoiding false positives on short identifiers that happen to share a prefix).

**No regex for prefix matching.** The main scan loop uses `str::find` on static prefix strings rather than compiled regex. This is faster for the common case of clean text with no credentials, and avoids the overhead of regex compilation.

**`contains_credentials` fast path.** Many tool outputs contain no credentials. The `contains_credentials` function does a fast substring scan (no allocation) so callers can skip the full `sanitize_credentials` pass when unnecessary.

**PEM block detection.** Private keys do not have short prefixes — they use `-----BEGIN ... PRIVATE KEY-----` delimiters. The sanitizer handles this as a special case, finding the BEGIN/END markers and replacing the entire block including all base64 content.

</Why>
