<ToolRef name="schedule_create" module="builtins/schedule.rs" crate="genesis-tools">

<What>

Creates a new scheduled prompt that Eve will execute automatically on a repeating cron schedule. You provide a standard 5-field cron expression and the prompt text to run. When the scheduler fires, Eve receives the prompt as if a user sent it, runs the full agent loop, and delivers the response to the configured destination platform.

This enables autonomous workflows: morning digests, periodic health checks, reminder messages, or any recurring task you would otherwise trigger manually.

</What>

<How>

## Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `cron` | `string` | Yes | — | 5-field cron expression: `minute hour day month weekday` |
| `prompt` | `string` | Yes | — | The prompt to execute when the schedule fires |
| `destination` | `string` | No | `"cli"` | Delivery platform: `cli`, `telegram`, `discord`, `slack`, `whatsapp` |

### Cron Expression Format

```
┌──────── minute (0-59)
│  ┌───── hour (0-23)
│  │  ┌── day of month (1-31)
│  │  │  ┌─ month (1-12)
│  │  │  │  ┌ weekday (0-6, Sunday=0)
│  │  │  │  │
*  *  *  *  *
```

Supported syntax: `*`, `N`, `*/N`, `N-M`, `N,M,P`.

## Example

```json
{
  "name": "schedule_create",
  "arguments": {
    "cron": "0 9 * * 1-5",
    "prompt": "Summarize the top 5 Hacker News stories and send me a digest",
    "destination": "telegram"
  }
}
```

**Response:**
```
Created schedule 'sched-a1b2c3d4': runs 'Summarize the top 5...' with cron '0 9 * * 1-5'
```

</How>

<Why>

The cron expression is validated immediately before writing to storage: if it does not have exactly 5 whitespace-separated fields, an error is returned. Actual cron parsing (ranges, steps, comma lists) is handled by the scheduler runtime in `genesis-core`, which polls every minute. The schedule ID is generated from the current nanosecond timestamp and process ID to avoid a UUID library dependency; it produces a sufficiently unique 8-character hex prefix in practice.

</Why>

</ToolRef>

<ToolRef name="schedule_list" module="builtins/schedule.rs" crate="genesis-tools">

<What>

Lists all configured schedules, showing each schedule's ID, enabled/disabled status, cron expression, destination platform, and prompt text. Takes no parameters.

</What>

<How>

## Parameters

None.

## Example

```json
{ "name": "schedule_list", "arguments": {} }
```

**Response:**
```
2 schedule(s):

- [enabled] sched-a1b2c3d4 | cron: 0 9 * * 1-5 | dest: telegram | prompt: Summarize HN stories
- [disabled] sched-e5f6g7h8 | cron: */30 * * * * | dest: cli | prompt: Check disk usage
```

</How>

<Why>

Schedules are stored in SQLite and loaded with a single `list_all` query. Disabled schedules are shown but not executed by the scheduler runtime. Operators can disable a schedule via the HTTP API (`PATCH /schedules/{id}/enabled`) without deleting it.

</Why>

</ToolRef>

<ToolRef name="schedule_delete" module="builtins/schedule.rs" crate="genesis-tools">

<What>

Permanently deletes a schedule by its ID. The schedule will no longer fire after deletion. If the ID does not exist, an error is returned rather than a silent no-op.

</What>

<How>

## Parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `id` | `string` | Yes | The schedule ID to delete (e.g. `sched-a1b2c3d4`) |

## Example

```json
{
  "name": "schedule_delete",
  "arguments": {
    "id": "sched-a1b2c3d4"
  }
}
```

**Response:**
```
Deleted schedule 'sched-a1b2c3d4'.
```

</How>

<Why>

The delete is a hard delete from SQLite with no soft-delete or recycle bin — use `schedule_list` first to confirm the right ID. The store's `delete` method returns a boolean indicating whether a row was actually removed; if false (no matching row), the tool surfaces an `ExecutionFailed` error so the model knows the operation did not succeed.

</Why>

</ToolRef>
