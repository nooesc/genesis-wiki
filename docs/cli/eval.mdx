<What>

## What is `genesis eval`?

The `eval` command group is Genesis's trajectory dataset management toolkit. Trajectories are JSON files produced by `genesis batch` — each one records a full agent run: the prompt, system message, every tool call and its output, model metadata, tags, and a success/failure outcome.

`genesis eval` gives you everything you need to work with these files: inspect individual trajectories, aggregate statistics across a directory, filter by quality or tags, convert to training formats, find duplicates, split into train/test sets, and run a full processing pipeline in a single command.

<How>

## Common Workflow

A typical trajectory dataset workflow looks like this:

```bash
# 1. Generate trajectories
genesis batch --input prompts.jsonl --output ./trajectories/ --auto-tag

# 2. Inspect the dataset
genesis eval stats ./trajectories/
genesis eval quality ./trajectories/ --min-score 0.6

# 3. Filter to high-quality successes
genesis eval filter ./trajectories/ --output ./filtered/ --min-quality 0.7 --success-only

# 4. Deduplicate
genesis eval deduplicate ./filtered/ --remove

# 5. Split into train/test
genesis eval split ./filtered/ --train ./train/ --test ./test/ --ratio 0.85

# 6. Export for fine-tuning
genesis eval export-chatml ./train/ --output train.jsonl
```

Or use the `pipeline` sub-command to run steps 3–6 in one invocation:

```bash
genesis eval pipeline ./trajectories/ --output ./ready/ \
  --validate --auto-tag --min-quality 0.7 --success-only \
  --format chatml --manifest
```

## Global Flags

All `eval` sub-commands respect the top-level `--json` flag for machine-readable output, and most accept `--recursive` to scan nested subdirectories.

<Why>

## Why a Separate Eval Toolkit?

The `eval` command group exists because training data curation is a distinct workflow from running the agent. Generating trajectories with `genesis batch` is fast; curating them — filtering noise, removing duplicates, balancing tags, splitting datasets — requires a different set of primitives.

Rather than piping `jq` invocations together, `eval` provides typed, tested operations that understand trajectory structure. Each sub-command reads and writes the same JSON format, so they compose naturally. The `pipeline` sub-command is the integrated path for users who want a reproducible, single-command dataset preparation step.

The quality scorer uses six dimensions (tool diversity, response length, success rate, turn efficiency, instruction following, error recovery) to assign a 0–1 score to each trajectory. This score is the primary filter for separating high-quality training examples from noise generated by exploratory or failed runs.
